# Generator Guide — spring-xpose processor

This document explains how the annotation processor pipeline works and how to add, modify, or extend generators.

---

## How the processor is registered

`ExposeEntityProcessor` is annotated with `@AutoService(Processor.class)` from `com.google.auto.service`. At build time, `auto-service` writes the fully-qualified class name into `META-INF/services/javax.annotation.processing.Processor` inside the `processor.jar`. The Java compiler discovers and invokes it automatically — no manual service-file editing needed.

```java
@SupportedAnnotationTypes("io.github.notablogger.springxpose.annotation.ExposeEntity")
@SupportedSourceVersion(SourceVersion.RELEASE_21)
@AutoService(Processor.class)
public class ExposeEntityProcessor extends AbstractProcessor { ... }
```

The processor runs in a **single round**: it iterates every `@ExposeEntity`-annotated `TypeElement`, builds an `EntityModel` for each, and fires all six generators. If a generator fails it emits a compile error on the entity class (not a silent crash).

---

## EntityModel — the shared data contract

Every generator receives an `EntityModel` record, not the raw `TypeElement`. This keeps generators decoupled from the javax.lang.model API and makes them unit-testable with plain Java records.

`EntityModel.parse(TypeElement, ProcessingEnvironment)` does all the reflection-like work:

| What it reads | Where it looks |
|---|---|
| `@ExposeEntity` attributes | `element.getAnnotation(ExposeEntity.class)` |
| `customMapper` class name | `AnnotationMirror` (avoids `MirroredTypeException`) |
| Scalar fields | `element.getEnclosedElements()` filtered to `FIELD` kind, non-static |
| `@Id` field | `jakarta.persistence.Id` presence on the field |
| `@Version` field | `jakarta.persistence.Version` presence on the field |
| Relation fields | `@ManyToOne`, `@OneToMany`, `@ManyToMany`, `@OneToOne` |
| Related entity type | `DeclaredType.getTypeArguments().get(0)` for collections, `DeclaredType.toString()` for single |
| Jakarta Validation annotations | `AnnotationMirror` filter on `jakarta.validation.*` prefix |
| Base path | `annotation.path()` or `EntityNameUtils.toBasePath(simpleName)` |

If no `@Id` field is found, `parse()` emits a compiler error and returns `null`. `ExposeEntityProcessor` skips null models and continues with the next entity.

## Generator anatomy

Every generator follows the same pattern:

```java
public class XyzGenerator {
    private final ProcessingEnvironment processingEnv;

    public XyzGenerator(ProcessingEnvironment processingEnv) {
        this.processingEnv = processingEnv;
    }

    public void generate(EntityModel model) {
        // 1. Build a JavaPoet TypeSpec
        TypeSpec typeSpec = TypeSpec.classBuilder(...)
            ...
            .build();

        // 2. Write it to the Filer
        JavaFile.builder(model.packageName() + ".generated", typeSpec)
            .addFileComment("Generated by spring-xpose — do not edit")
            .build()
            .writeTo(processingEnv.getFiler());
    }
}
```

All generated types land in the `<entity-package>.generated` sub-package — this keeps generated sources clearly separated from hand-written code.

## Generator responsibilities

### `RepositoryGenerator`
- Generates `<Entity>Repository extends JpaRepository<Entity, IdType>`
- Annotated with `@Repository` and marked as a Spring bean
- No custom query methods — Spring Data's defaults cover all five `Operation` values

### `DtoGenerator`
- Generates `<Entity>Dto` — the API response shape
- Includes all scalar fields **except** `ignoredFields`
- `@Id` field is always included
- `@ManyToOne` / `@OneToOne` relations:
  - `ALWAYS_OBJECT` → field of the related entity type (full object)
  - `ALWAYS_IDS` or `IDS_FOR_LIST_OBJECT_FOR_SINGLE` → `Long <fieldName>Id`
- `@OneToMany` / `@ManyToMany` → always omitted (prevents circular serialisation)
- Primitive types are boxed (e.g. `long` → `Long`) so they can be null in JSON

### `RequestDtoGenerator`
- Generates `<Entity>RequestDto` — the API request body shape
- Never includes `@Id` or `ignoredFields` fields
- Copies Jakarta Validation annotations (`@NotBlank`, `@Positive`, etc.) from entity fields
- Relations are always `Long <fieldName>Id` regardless of `relationMode`
- `@OneToMany` / `@ManyToMany` always omitted

### `MapperGenerator`
- Generates `<Entity>Mapper` — a MapStruct interface
- Four methods:
  - `toDto(Entity) → Dto` — with `@Mapping(source="rel.id", target="relId")` for ID-mode relations
  - `toDtoList(List<Entity>) → List<Dto>`
  - `toEntity(RequestDto) → Entity` — ignores id, `@Version`, `ignoredFields`, and relation fields (controller handles those via `EntityManager.getReference`)
  - `updateEntity(RequestDto, @MappingTarget Entity) → void` — same ignores; used by PUT to overlay only request fields onto the loaded entity
- If `customMapper` is set, omits `@Mapper(componentModel = "spring")` so MapStruct skips auto-generation; the interface is still generated as a contract

### `RestControllerGenerator`
- Generates `<Entity>Controller` — a `@RestController` at `/api/<basePath>`
- Conditionally generates only the methods listed in `expose`
- `findAll` returns `List<Dto>` by default; `Page<Dto>` when `pageable = true`
- `create` / `update` / `delete` are `@Transactional`
- `update` uses load-then-merge: fetches the entity first, calls `mapper.updateEntity()`, then saves
- Relation IDs are resolved via `EntityManager.getReference()` (proxy, no SELECT)
- All methods carry full OpenAPI annotations (`@Tag`, `@Operation`, `@ApiResponse`, `@Parameter`)
- Sets `SerializationContext` (ThreadLocal) before returning to control relation serialisation mode

### `SecurityConfigurerGenerator`
- Generates `<Entity>SecurityConfigurer` — a `@Configuration` with a `@Bean SecurityFilterChain`
- Uses `securityMatcher` to scope the chain to `/api/<basePath>/**`
- `@Order` value is deterministically derived from `entity.hashCode()` (range 100–999)
- Builds a single `authorizeHttpRequests` block:
  - `readRoles` (or `roles`) applied to GET matchers
  - `writeRoles` (or `roles`) applied to POST / PUT / DELETE matchers
  - Fallback: `authenticated()` when no roles specified
  - `anyRequest().denyAll()` — fail-closed

## Adding a new generator

1. **Create the class** in `processor/src/main/java/.../processor/generator/`:

```java
public class MyNewGenerator {
    private final ProcessingEnvironment processingEnv;

    public MyNewGenerator(ProcessingEnvironment processingEnv) {
        this.processingEnv = processingEnv;
    }

    public void generate(EntityModel model) {
        TypeSpec typeSpec = TypeSpec.classBuilder(model.entitySimpleName() + "MyNew")
            .addModifiers(Modifier.PUBLIC)
            .addJavadoc("Generated by spring-xpose — do not edit\n")
            // ... build your type
            .build();

        try {
            JavaFile.builder(model.packageName() + ".generated", typeSpec)
                .addFileComment("Generated by spring-xpose — do not edit")
                .build()
                .writeTo(processingEnv.getFiler());
        } catch (IOException e) {
            throw new RuntimeException("Failed to write MyNew for " + model.entitySimpleName(), e);
        }
    }
}
```

2. **Invoke it** in `ExposeEntityProcessor.process()` alongside the existing generators:

```java
try { new MyNewGenerator(processingEnv).generate(model); }
catch (Exception e) { error(entityClass, "my-new", e); }
```

3. **Add to `EntityModel`** if the generator needs data that isn't already there — add a field to the record and populate it in `parse()`.

4. **Write tests** — see [`generator-guide.md`](generator-guide.md#testing-generators) below.

5. **Document** the new artifact in [`generated-artifacts.md`](generated-artifacts.md).

## Adding a new `@ExposeEntity` attribute

1. Add the attribute to `ExposeEntity.java` in the `annotations` module with a Javadoc comment and a sensible default:

```java
/**
 * When {@code true}, adds an X-Request-ID header to all responses.
 */
boolean requestId() default false;
```

2. Parse it in `EntityModel.parse()` — add a field to the record and read from the annotation:

```java
// in the record header:
boolean requestId,

// in parse():
annotation.requestId(),   // added to the constructor call
```

3. Use the flag in the relevant generator(s).

4. Update `README.md` (attribute table) and `docs/tech/generated-artifacts.md`.

## Testing generators

Tests live in `processor/src/test/java/.../processor/`. They use `com.google.testing.compile` to compile source strings in-memory through the real APT pipeline and then assert on the generated output.

### Pattern

```java
@Test
void generatesRepositoryForBasicEntity() {
    Compilation compilation = Compiler.javac()
        .withProcessors(new ExposeEntityProcessor())
        .compile(JavaFileObjects.forSourceString(
            "com.example.Product",
            """
            package com.example;
            import io.github.notablogger.springxpose.annotation.*;
            import jakarta.persistence.*;
            @Entity
            @ExposeEntity
            public class Product {
                @Id private Long id;
                private String name;
                public Long getId() { return id; }
                public String getName() { return name; }
            }
            """));

    assertThat(compilation).succeeded();
    assertThat(compilation)
        .generatedSourceFile("com.example.generated.ProductRepository")
        .contentsAsUtf8String()
        .contains("extends JpaRepository<Product, Long>");
}
```

### What to assert

- `compilation.succeeded()` / `compilationWithoutError()` — always first
- `.generatedSourceFile(fqcn)` — the file exists
- `.contentsAsUtf8String().contains(...)` — specific methods, annotations, or field names are present
- For negative cases: `doesNotContain(...)` — e.g., `@Mapper` absent when `customMapper` is set
- For compiler errors: `compilationWithoutWarnings()` or `hadErrorContaining(...)`

### Running tests

```bash
./gradlew :processor:test
# with continuous output:
./gradlew :processor:test --info
```

## Utility — `EntityNameUtils`

`EntityNameUtils` is a stateless utility class with three methods:

| Method | Input | Output | Example |
|---|---|---|---|
| `pluralize(name)` | `"Product"` | `"products"` | handles -s, -x, -ch, -y → -ies, person → people |
| `toBasePath(simpleName)` | `"Category"` | `"categories"` | delegates to `pluralize`, lowercased |
| `toLowerCamel(name)` | `"Product"` | `"product"` | lowercases first character |

`pluralize` covers the most common English irregular forms. For entities with genuinely unusual plurals, set `path` explicitly in `@ExposeEntity(path = "my-custom-path")`.
